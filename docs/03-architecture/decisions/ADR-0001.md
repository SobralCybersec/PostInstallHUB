# ADR-0001: Single curl Entry Point with OS Auto-Detection

- Status: accepted
- Date: 2026-07-13
- Deciders: Matheus
- Supersedes: none
- Related requirements: FR-001 (OS detection), SCOPE.md

## Context

Users need to run a post-install script on a fresh operating system with
minimal friction. A fresh machine has almost nothing on it: no package manager
configured, no language runtimes, no knowledge of which specific script to run
for their distro. Arch users and Ubuntu users face different package managers,
different package names, and different configuration paths. Without a
centralized entry point, the documentation would need to list a separate curl
command (or a separate clone-and-run instruction) for every supported platform,
and users would need to correctly identify their own distro before running
anything.

The alternative — per-distro scripts with per-distro instructions — also means
that the project's shareable one-liner (the thing you paste into a README or
tweet) either doesn't exist or is distro-specific, significantly reducing the
project's utility as a "just run this" tool.

## Decision drivers

- Zero prerequisite knowledge from the user: they should not need to know their
  distro's codename or which script file corresponds to it.
- One shareable command works everywhere in the supported matrix.
- New distro support must not change the user-facing command.
- The routing logic must be inspectable and auditable in a single file.

## Considered options

1. **Single `install.sh` with auto-detection (this decision):** One entry
   point reads `/etc/os-release` (Linux) or detects the Windows environment
   and routes to the correct distro-specific handler automatically.

2. **Per-distro curl commands:** Separate scripts per distro, separate
   instructions per distro. Users run `curl .../ubuntu.sh`, `curl .../arch.sh`,
   etc., based on their own knowledge of their OS.

3. **Ansible playbook as entry point:** One `ansible-playbook` command that
   handles distro differences internally through Ansible's module abstraction
   and `ansible_facts`.

## Decision

We will provide a single `install.sh` entry point that reads `/etc/os-release`
(Linux) or equivalent environment signals (Windows CMD) and routes to the
correct distro-specific script automatically. The user-facing command is always:

```bash
curl -fsSL https://raw.githubusercontent.com/<owner>/PostInstallHUB/main/install.sh | bash
```

Distro handlers live in `distros/` (e.g., `distros/ubuntu.sh`,
`distros/arch.sh`, `distros/fedora.sh`, `distros/omarchy.sh`). The entry point
sources the correct handler after detection; it does not duplicate
installation logic.

## Rationale

**Option 1** is the only choice that satisfies all four decision drivers. The
detection logic is a well-understood pattern (`source /etc/os-release; case
$ID in ...`), robust across all supported distros, and readable in under 30
lines. Adding a new distro is a one-file addition plus a new case branch in
`install.sh` — the user-facing command never changes.

**Option 2** fails the first and second drivers outright. Users must know their
distro, remember which URL to use, and documentation must maintain multiple
commands. Sharing the project means sharing the right link for the recipient's
OS, which is friction that defeats the purpose.

**Option 3** fails on the most fundamental constraint: a fresh OS does not have
Ansible. Installing Ansible before running the post-install script is circular
— it is itself a post-install step. Ansible is also a heavyweight dependency
for what is ultimately a few dozen shell commands.

## Consequences

### Positive

- One command covers all supported distros: easier to document, share, and
  remember.
- Distro-specific logic is fully isolated in `distros/`; `install.sh` stays
  thin and auditable.
- Adding Omarchy, Fedora, or any future distro does not change the user
  experience at all.
- CI can test detection logic in isolation by mocking `/etc/os-release`.

### Negative

- `install.sh` must be updated whenever a new distro is added to the supported
  matrix; it is a choke point for the detection routing table.
- There is a small layer of indirection — users who want to run only the Arch
  script can do so directly (`bash distros/arch.sh`), but the default entry
  point adds one routing step.
- If `/etc/os-release` is absent or malformed (unusual but possible on
  minimal container images), detection fails and the user must run the
  distro-specific script directly.

### Risks

- **Detection false positives:** a distro that sets `ID=ubuntu` but diverges
  significantly (e.g., a heavily modified Ubuntu spin) may pass detection but
  fail installation. Mitigated by testing against clean images in CI and by the
  non-destructive nature of the script (it only installs, never removes).
- **Scope creep in `install.sh`:** as distros multiply, the routing table
  grows. Mitigated by keeping `install.sh` as a pure router — no installation
  logic lives there; it only detects and delegates.

## Validation

Considered successful when:

- `bash install.sh` on a clean Ubuntu 24.04, Arch, Fedora, and Omarchy image
  each reach the correct distro handler without user input.
- The same `curl | bash` one-liner is the only user-facing command documented
  in `README.md`.
- No `{{PLACEHOLDER}}` values remain in this ADR.

Review at the first working multi-distro integration test run.

## Revisit when

- The supported distro count exceeds ~10 and the routing table in `install.sh`
  becomes unwieldy — consider generating it from a manifest file.
- A user-facing distro override flag is requested (`--distro arch`) — add it
  to `install.sh` without changing the default auto-detect behavior.
- Windows support matures to the point where a separate `install.cmd` entry
  point is cleaner than a shared `install.sh` with Windows detection.
